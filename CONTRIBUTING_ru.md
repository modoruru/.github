# Контрибьютинг
## Основное руководоство

### Не создавайте PR для форматирования или "очистки" кода.
Если вы отправляете PR с изменениями в коде, можете исправить форматирование в области изменений, но, пожалуйста, не тратьте время всех на бессмысленные изменения.
Ни у кого нет времени рассматривать PR на изменение 1-2 строк.

### Не отправляйте изменений сделанных ИИ
Задавать вопросы ИИ и использовать эту информацию для написания кода - хорошо. Использовать ИИ для написания кода - Нет.
Вы будете заблокированы если будете отправять ИИ код в наши репозитории.

## Руководоство по стилю кода
- Используйте единственное число для пакетов, содержащих базовые интерфейсы или абстрактные классы. 
- Используйте множественное число только для пакетов, содержащих конкретные реализации.
- Избегайте использования префиксов `get`, `set` и `is` в названиях методов. Сигнатура метода и контекст должны ясно указывать на его назначение.
- Помечайте каждый класс как final, если он явно не предназначен для наследования.
- Всегда делайте утилитные классы `final` с `private` конструктором, чтобы предотвратить создание экземпляров.
- Не добавляйте внешние зависимости для функциональности, которая уже доступна в текущих зависимостях.
- Если блок кода не очень большой или не используется более чем в 1-2 местах, не следует выносить его в отдельный метод.

## Руководоство по Pull Request'ам
- Весь код содержащийся в PR должен следовать выше указанным требованиям, в противном случае мы попросим вас исправить это/откажем в merge'е.
- Если вы публикуете новый функционал, по необходимости напишите тесты, чтобы мы убедились в работе вашего изменения.
